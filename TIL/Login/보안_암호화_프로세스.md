## 1. 보안 코드 구조

```
lib/core/security/
  ├── password_hasher.dart   ← 비밀번호 단방향 해시 + 검증
  └── field_encryptor.dart   ← 개인정보 필드 양방향 암호화/복호화
```

`core/security/`로 분리한 이유:
- 보안 정책 코드와 일반 유틸(날짜 포맷, UUID 등)을 명확히 구분
- 보안 관련 코드가 늘어날 때 한 곳에서 관리 가능

## 2. 단방향 vs 양방향

| 구분 | 방식 | 복호화 | 사용 필드 |
|---|---|---|---|
| 단방향 | SHA-256 해시 | ❌ 불가 | `auth_pwd` (비밀번호) |
| 양방향 | AES-256 암호화 | ✅ 가능 | `emp_addr`, `tel_no`, `cel_no`, `e_mail` |

### 왜 용도가 다른가?

```
비밀번호  →  "맞는지 비교"만 하면 됨
             원문을 볼 필요가 없음
             복호화 불가 → 유출돼도 원문 알 수 없음 (더 안전)

개인정보  →  화면에 표시해야 하므로 반드시 복호화 필요
             AES 키가 있어야만 원문 복원 가능
```

## 3. password_hasher.dart — SHA-256 단방향 해시

### 핵심 개념

```dart
class PasswordHasher {
  PasswordHasher._(); // private 생성자 → 인스턴스 생성 불가

  // 평문 → SHA-256 해시 문자열
  static String hash(String plainPassword) {
    final bytes = utf8.encode(plainPassword); // 문자열 → 바이트 배열
    return sha256.convert(bytes).toString();  // 바이트 → 해시 문자열
  }

  // 평문과 저장된 해시 비교 (대소문자 구분)
  static bool verify(String plainPassword, String hashedPassword) {
    return hash(plainPassword) == hashedPassword;
  }
}
```

### 동작 원리

```
로그인 시도: 'admin52345234'
      ↓ hash()
  '76b39e063860...'
      ↓ == 비교
DB 저장값: '76b39e063860...'
      ↓
  true → 로그인 성공
```

원문끼리 비교하는 게 아니라 **입력값도 똑같이 해시한 뒤 해시끼리 비교**합니다.  
DB에 원문이 없어도 검증이 가능하고, DB가 유출돼도 원문을 알 수 없습니다.

### verify()와 hash()의 차이

| 메서드 | 역할 | 현재 사용 여부 |
|---|---|---|
| `hash()` | 평문 → 해시 변환 | ✅ 사용 중 (DB 저장 및 SQL 비교) |
| `verify()` | 평문과 해시 일치 확인 | 대기 중 (SQL 없이 코드에서 직접 비교할 때 사용) |

현재는 SQL WHERE 조건에서 `hash(입력값)`을 DB 저장값과 비교하므로 `verify()`를 별도로 호출하지 않습니다.

### 주의 사항

```
대소문자 구분 → trim()만 적용, toLowerCase() 사용 금지
'Admin' ≠ 'admin' → 해시값이 달라 로그인 실패
```

## 4. field_encryptor.dart — AES-256 양방향 암호화

### 핵심 개념

```dart
class FieldEncryptor {
  FieldEncryptor._();

  // AES-256: 반드시 32바이트(32글자) 키 필요
  static final Key _key = Key.fromUtf8('WmsPdaKMarket2024SecureKey!@#$%&');
  //                                    ↑ 정확히 32글자 = 256비트

  static final _encrypter = Encrypter(AES(_key, mode: AESMode.cbc));
}
```

### AES Key 역할

`Key.fromUtf8(...)` 안의 문자열이 **암호화/복호화에 사용하는 열쇠**입니다.

```
암호화: '010-1234-5678' + 키 → 'Abc123:Def456'
복호화: 'Abc123:Def456' + 같은 키 → '010-1234-5678' ✅
복호화: 'Abc123:Def456' + 다른 키 → 쓰레기값 or 에러  ❌
```

키 길이와 AES 종류:

| AES 종류 | 키 길이 |
|---|---|
| AES-128 | 16글자 (128비트) |
| AES-192 | 24글자 (192비트) |
| AES-256 | **32글자 (256비트)** ← 현재 사용 |

### IV (초기화 벡터) 역할

```dart
static String encrypt(String value) {
  final iv = IV.fromSecureRandom(16); // 매번 랜덤 IV 생성
  final encrypted = _encrypter.encrypt(value, iv: iv);
  return '${iv.base64}:${encrypted.base64}'; // IV와 암호문을 함께 저장
}
```

IV를 매번 랜덤으로 생성하는 이유:

```
IV 고정 시 → 같은 값은 항상 같은 암호문
  '010-1234-5678' → 항상 'Abc123:Def456'
  → 패턴 분석으로 추측 가능 (보안 취약)

IV 랜덤 시 → 같은 값도 매번 다른 암호문
  '010-1234-5678' → 첫 번째 암호화: 'Abc123:Def456'
  '010-1234-5678' → 두 번째 암호화: 'Xyz789:Uvw012'
  → 패턴 분석 불가 (보안 강화)
```

저장 형식 `"IV:암호문"` 으로 저장하는 이유:
- 복호화 시 어떤 IV로 암호화했는지 알아야 하기 때문

### null 허용 버전

```dart
// emp_addr, tel_no 등은 값이 없을 수 있어서 null 허용 버전 별도 제공
static String? encryptNullable(String? value) {
  if (value == null || value.isEmpty) return null;
  return encrypt(value);
}
```

## 5. DB 저장 시 암호화 적용 흐름

```
WMS API 응답 (평문)
  emp_addr: '하노이 바딘구 123번지'
  tel_no:   '024-1234-5678'
  auth_pwd: 'admin52345234'
        ↓
auth_local_datasource.upsertUser()
        ↓
  auth_pwd  → PasswordHasher.hash()    → '76b39e...'         (단방향)
  emp_addr  → FieldEncryptor.encrypt() → 'Abc123:Def456'     (양방향)
  tel_no    → FieldEncryptor.encrypt() → 'Xyz789:Uvw012'     (양방향)
        ↓
SQLite wms_user 테이블에 암호화된 값으로 저장
```

```
로그인 시 DB 조회
        ↓
auth_local_datasource._toModel()
        ↓
  emp_addr  → FieldEncryptor.decrypt() → '하노이 바딘구 123번지' (복호화)
  tel_no    → FieldEncryptor.decrypt() → '024-1234-5678'        (복호화)
  auth_pwd  → 복호화 없음 (hash 비교는 SQL WHERE에서 처리)
        ↓
UserModel (평문) → 화면 표시
```

## 6. 보안 강도 비교

| 항목 | 현재 구현 | 위험도 |
|---|---|---|
| 비밀번호 저장 | SHA-256 해시 | 낮음 — 복호화 불가 |
| 개인정보 저장 | AES-256 암호화 | 중간 — 키 탈취 시 노출 가능 |
| AES 키 관리 | **앱 내 하드코딩** | **높음 — 개선 필요** |

> 작성일: 2026-04-08

## 1. 오류처리 구조 개요

오류처리는 두 파일로 역할을 분리합니다.

| 파일 | 위치 | 역할 |
|---|---|---|
| `app_exception.dart` | `lib/core/error/` | 오류 **종류(타입) 정의** — 빈 상자 |
| `api_error_handler.dart` | `lib/core/error/` | 오류 **판단 및 변환** — 상자에 담는 역할 |

---

## 2. app_exception.dart — 오류 타입 정의

```dart
sealed class AppException implements Exception {
  final String message;
  const AppException(this.message);
}

class NetworkException        extends AppException { ... }  // 네트워크 연결 불가 / 타임아웃
class AuthException           extends AppException { ... }  // 인증 실패 (401, 403)
class ServerException         extends AppException { ... }  // 서버 오류 (5xx 등)
class OfflineFallbackException extends AppException { ... } // 오프라인 + 로컬 캐시 없음
```

### sealed 키워드의 의미

`sealed`는 **같은 파일 안에서만 상속을 허용**하는 키워드입니다.

| | `abstract class` | `sealed class` |
|---|---|---|
| 상속 가능 범위 | 어디서든 | 같은 파일만 |
| 서브타입 파악 | 컴파일러 모름 | 컴파일러가 정확히 앎 |
| switch 완전성 검사 | ❌ | ✅ |

**장점 — switch에서 케이스 누락 시 컴파일 경고:**
```dart
switch (exception) {
  case NetworkException e  => '네트워크 오류',
  case AuthException e     => '인증 오류',
  case ServerException e   => '서버 오류',
  // OfflineFallbackException 빠뜨리면 컴파일러가 경고 발생 → 실수 방지
}
```

## 3. api_error_handler.dart — 오류 판단 및 변환

```dart
class ApiErrorHandler {
  ApiErrorHandler._(); // private 생성자 → 인스턴스 생성 불가, static 메서드만 사용
  
  static AppException handle(DioException e) { ... }
}
```

### private 생성자 `._()` 의 의미

유틸 클래스처럼 static 메서드만 쓸 때, 불필요한 인스턴스 생성을 막기 위해 사용합니다.

```dart
ApiErrorHandler.handle(e);          // ✅ 의도한 사용
final handler = ApiErrorHandler();  // ❌ 컴파일 에러 → 실수 방지
```

## 5. 오류 종류별 분기 처리

```dart
// api_error_handler.dart
static AppException handle(DioException e) {
    if (_isNetworkError(e)) {
      return const NetworkException();   // 서버에 아예 연결 못한 경우
    }

    return switch (statusCode) {
      401 => const AuthException('아이디 또는 비밀번호가 올바르지 않습니다.'),
      403 => const AuthException('접근 권한이 없습니다.'),
      404 => const ServerException('요청한 리소스를 찾을 수 없습니다.'),
      _   => ServerException('서버 오류가 발생했습니다. ($statusCode)'),
    };
}

// _isNetworkError: 아래 4가지 타입이면 네트워크 오류로 판단
// - connectionError   : 서버 자체에 연결 불가
// - connectionTimeout : 연결 시도 중 시간 초과
// - receiveTimeout    : 응답 대기 중 시간 초과
// - sendTimeout       : 요청 전송 중 시간 초과
```

## 6. Repository에서의 오류 분기 — 로컬 폴백 판단

```dart
// auth_repository.dart
try {
  final user = await _remote.login(id, pw);  // API 호출
  ...
} on NetworkException {
  // 네트워크 오류 → 로컬 캐시 폴백 허용
  final cached = await _local.login(id, pw);
  if (cached != null) return cached;         // 캐시 있으면 로그인 성공
  throw const OfflineFallbackException();    // 캐시도 없으면 실패
  
} on AppException catch (e) {
  // 서버가 응답했지만 인증 거부(401) 등 → 로컬 폴백 없이 즉시 오류
  rethrow;
}
```

### 로컬 폴백 허용 여부 결정 기준

| 오류 종류 | 로컬 폴백 | 이유 |
|---|---|---|
| `NetworkException` | ✅ 허용 | 서버를 만나지 못한 것 → 로컬에서 시도 가능 |
| `AuthException` (401) | ❌ 차단 | 서버가 "틀렸다"고 확인함 → 로컬도 막아야 보안 유지 |
| `ServerException` (5xx) | ❌ 차단 | 서버 문제 → 사용자에게 오류 안내 |

## 7. 최종 화면 표시

```dart
// login_provider.dart
} on Exception catch (e) {
    state = state.copyWith(
      isLoading: false,
      errorMessage: e.toString().replaceFirst('Exception: ', ''),
    );
}
```

`AppException.toString()`은 `message`를 반환하므로, 위에서 정의한 한국어 메시지가 그대로 화면에 표시됩니다.

## 8. 전체 비유 요약

```
DioException    →  Dio가 던진 날것의 오류 (영어, 기술적)
        ↓
ApiErrorHandler →  "이건 네트워크 문제야, 이건 인증 문제야" 판단
        ↓
AppException    →  한국어 메시지를 담은 타입별 오류 상자
        ↓
Repository      →  NetworkException이면 로컬 폴백, 나머지는 그대로 throw
        ↓
Provider        →  errorMessage로 화면에 표시
```

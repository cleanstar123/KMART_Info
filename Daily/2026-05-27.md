## 1. CustomUserDetailsService — DB 조회 이중 호출 제거

> [!info] 배경
> - `CustomUserDetailsService.loadUserByUsername()`는 Spring Security가 JWT 인증 필터에서 **매 API 요청마다** 호출하는 메서드
> - 기존에는 DB를 2번 조회하고 있었음
>   - `selectUser(userId)` → userId + password 조회
>   - `selectUserInfo(userId)` → role_id(권한) 조회

> [!warning] 문제
> - PDA가 API를 호출할 때마다 DB hit이 2회 발생
> - 베트남 현지 네트워크 지연 환경에서 DB(외부 IP `210.219.229.46`)까지의 왕복 지연이 누적됨
> - 로그인 1회에 총 3번 `selectUserInfo`가 호출되는 구조였음
>   - `loadUserByUsername` 내부에서 1번 + `AuthServiceImpl.login()`에서 1번 (웹/PDA 공통)

> [!success] 해결
> - `selectUser`를 `userId + password + role_id`를 **한 번에 조회**하는 쿼리로 통합
>   - 기존 `selectUser`는 `userId + password`만 반환했으나, `role_id`까지 포함하도록 SQL 수정
> - `CustomUserDetailsService`에서 `selectUserInfo` 추가 호출 제거
> - DB 조회 횟수: **2회 → 1회** (매 인증 요청 기준)
>
> ```java
> // CustomUserDetailsService.java — 변경 후
> public UserDetails loadUserByUsername(String userId) {
>     UserInfoDTO user = authMapper.selectUser(userId); // 단일 쿼리로 통합
>     AuthCode authCode = AuthCode.valueOf(user.getAuthCode());
>     return new User(user.getUserId(), user.getPassword(),
>             List.of(new SimpleGrantedAuthority(authCode.getRole())));
> }
> ```
>
> ```xml
> <!-- AuthMapper.xml — selectUser 쿼리 변경 -->
> <select id="selectUser" resultType="com.kmarket.wms.auth.dto.UserInfoDTO">
>     SELECT user_id,
>            pswd    AS password,
>            role_id AS auth_code
>     FROM wms.cm_user
>     WHERE user_id = #{userId} AND use_yn = 'Y'
> </select>
> ```

> [!tip] 배운 점
> - `loadUserByUsername`은 로그인 시뿐만 아니라 **인증된 모든 API 요청**마다 실행됨
> - 단순해 보이는 메서드라도 호출 빈도가 높으면 DB 조회 비용이 누적됨

## 2. Caffeine 인메모리 캐시 도입

> [!info] 배경
> - 위 1번 개선 이후에도 매 API 요청마다 DB 조회는 1회 발생
> - 동일한 userId로 반복 요청이 오는 경우(PDA 작업 중 연속 API 호출) DB 왕복 비용이 매번 발생
> - 베트남 현지 DB 서버(`210.219.229.46:7600`) 접속 지연 문제로 성능 개선 필요

> [!warning] 문제
> - 캐시가 전혀 없는 구조 → 동일 사용자가 1시간 동안 100번 API 호출 시 DB 조회 100회
> - `GET /auth/users` (PDA 앱 기동 시 전체 사용자 동기화) 도 매번 DB에서 전체 조회

> [!success] 해결
> **추가된 파일 및 변경 내용**
>
> ① `pom.xml` — 의존성 2개 추가
> ```xml
> <dependency>
>     <groupId>org.springframework.boot</groupId>
>     <artifactId>spring-boot-starter-cache</artifactId>
> </dependency>
> <dependency>
>     <groupId>com.github.ben-manes.caffeine</groupId>
>     <artifactId>caffeine</artifactId>
> </dependency>
> ```
>
> ② `config/CacheConfig.java` — 신규 생성
> - 캐시 이름: `userDetails`(인증 사용자 정보), `userList`(PDA 동기화 사용자 목록)
> - TTL: 30분, 최대 항목 수: 500건
> ```java
> @Configuration
> @EnableCaching
> public class CacheConfig {
>     @Bean
>     public CacheManager cacheManager() {
>         CaffeineCacheManager manager = new CaffeineCacheManager("userDetails", "userList");
>         manager.setCaffeine(
>             Caffeine.newBuilder()
>                 .expireAfterWrite(30, TimeUnit.MINUTES)
>                 .maximumSize(500)
>         );
>         return manager;
>     }
> }
> ```
>
> ③ `CustomUserDetailsService.java` — `@Cacheable` 추가
> - userId를 키로 `UserDetails`를 30분간 메모리에 보관
> - 동일 사용자의 반복 API 호출 시 DB 조회 없이 메모리에서 즉시 반환
> ```java
> @Cacheable(value = "userDetails", key = "#userId")
> public UserDetails loadUserByUsername(String userId) { ... }
> ```
>
> ④ `AuthServiceImpl.java` — `@Cacheable` + `@CacheEvict` 추가
> - `getUserList()`: 전체 사용자 목록을 30분간 캐시 (PDA 기동 시 반복 요청 대비)
> - `join()`: 신규 사용자 등록 시 `userList` 캐시 자동 삭제 → 다음 동기화 때 최신 데이터 반영
> ```java
> @Cacheable(value = "userList", key = "'all'")
> public List<UserInfoDTO> getUserList() { ... }
>
> @CacheEvict(value = "userList", key = "'all'")
> public void join(AuthRequest authRequest) { ... }
> ```

> [!tip] 배운 점
> - Caffeine은 Redis 없이 JVM 메모리에서 동작하므로 별도 인프라 없이 즉시 적용 가능
> - `@Cacheable`은 Spring AOP 기반이므로 **같은 클래스 내 자기 호출(self-invocation)에는 동작하지 않음** — 반드시 외부 빈에서 호출해야 캐시가 적용됨
> - `@CacheEvict`로 데이터 변경 시점에 캐시를 무효화해야 정합성이 유지됨

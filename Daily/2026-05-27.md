## 1. CustomUserDetailsService — DB 조회 이중 호출 제거

> [!info] 배경
> - `CustomUserDetailsService.loadUserByUsername()`는 Spring Security가 JWT 인증 필터에서 **매 API 요청마다** 호출하는 메서드
> - 기존에는 DB를 2번 조회하고 있었음
>   - `selectUser(userId)` → userId + password 조회
>   - `selectUserInfo(userId)` → role_id(권한) 조회

> [!warning] 문제
> - PDA가 API를 호출할 때마다 DB hit이 2회 발생
> - 베트남 현지 네트워크 지연 환경에서 DB(외부 IP `210.219.229.46`)까지의 왕복 지연이 누적됨
> - 로그인 1회에 총 3번 `selectUserInfo`가 호출되는 구조였음
>   - `loadUserByUsername` 내부에서 1번 + `AuthServiceImpl.login()`에서 1번 (웹/PDA 공통)

> [!success] 해결
> - `selectUser`를 `userId + password + role_id`를 **한 번에 조회**하는 쿼리로 통합
>   - 기존 `selectUser`는 `userId + password`만 반환했으나, `role_id`까지 포함하도록 SQL 수정
> - `CustomUserDetailsService`에서 `selectUserInfo` 추가 호출 제거
> - DB 조회 횟수: **2회 → 1회** (매 인증 요청 기준)
>
> ```java
> // CustomUserDetailsService.java — 변경 후
> public UserDetails loadUserByUsername(String userId) {
>     UserInfoDTO user = authMapper.selectUser(userId); // 단일 쿼리로 통합
>     AuthCode authCode = AuthCode.valueOf(user.getAuthCode());
>     return new User(user.getUserId(), user.getPassword(),
>             List.of(new SimpleGrantedAuthority(authCode.getRole())));
> }
> ```
>
> ```xml
> <!-- AuthMapper.xml — selectUser 쿼리 변경 -->
> <select id="selectUser" resultType="com.kmarket.wms.auth.dto.UserInfoDTO">
>     SELECT user_id,
>            pswd    AS password,
>            role_id AS auth_code
>     FROM wms.cm_user
>     WHERE user_id = #{userId} AND use_yn = 'Y'
> </select>
> ```

> [!tip] 배운 점
> - `loadUserByUsername`은 로그인 시뿐만 아니라 **인증된 모든 API 요청**마다 실행됨
> - 단순해 보이는 메서드라도 호출 빈도가 높으면 DB 조회 비용이 누적됨

## 2. Caffeine 인메모리 캐시 도입

> [!info] 배경
> - 위 1번 개선 이후에도 매 API 요청마다 DB 조회는 1회 발생
> - 동일한 userId로 반복 요청이 오는 경우(PDA 작업 중 연속 API 호출) DB 왕복 비용이 매번 발생
> - 베트남 현지 DB 서버(`210.219.229.46:7600`) 접속 지연 문제로 성능 개선 필요

> [!warning] 문제
> - 캐시가 전혀 없는 구조 → 동일 사용자가 1시간 동안 100번 API 호출 시 DB 조회 100회
> - `GET /auth/users` (PDA 앱 기동 시 전체 사용자 동기화) 도 매번 DB에서 전체 조회

> [!success] 해결
> **추가된 파일 및 변경 내용**
>
> ① `pom.xml` — 의존성 2개 추가
> ```xml
> <dependency>
>     <groupId>org.springframework.boot</groupId>
>     <artifactId>spring-boot-starter-cache</artifactId>
> </dependency>
> <dependency>
>     <groupId>com.github.ben-manes.caffeine</groupId>
>     <artifactId>caffeine</artifactId>
> </dependency>
> ```
>
> ② `config/CacheConfig.java` — 신규 생성
> - 캐시 이름: `userDetails`(인증 사용자 정보), `userList`(PDA 동기화 사용자 목록)
> - TTL: 30분, 최대 항목 수: 500건
> ```java
> @Configuration
> @EnableCaching
> public class CacheConfig {
>     @Bean
>     public CacheManager cacheManager() {
>         CaffeineCacheManager manager = new CaffeineCacheManager("userDetails", "userList");
>         manager.setCaffeine(
>             Caffeine.newBuilder()
>                 .expireAfterWrite(30, TimeUnit.MINUTES)
>                 .maximumSize(500)
>         );
>         return manager;
>     }
> }
> ```
>
> ③ `CustomUserDetailsService.java` — `@Cacheable` 추가
> - userId를 키로 `UserDetails`를 30분간 메모리에 보관
> - 동일 사용자의 반복 API 호출 시 DB 조회 없이 메모리에서 즉시 반환
> ```java
> @Cacheable(value = "userDetails", key = "#userId")
> public UserDetails loadUserByUsername(String userId) { ... }
> ```
>
> ④ `AuthServiceImpl.java` — `@Cacheable` + `@CacheEvict` 추가
> - `getUserList()`: 전체 사용자 목록을 30분간 캐시 (PDA 기동 시 반복 요청 대비)
> - `join()`: 신규 사용자 등록 시 `userList` 캐시 자동 삭제 → 다음 동기화 때 최신 데이터 반영
> ```java
> @Cacheable(value = "userList", key = "'all'", condition = "#since == null")
> public List<UserInfoDTO> getUserList(String since) { ... }
>
> @CacheEvict(value = "userList", key = "'all'")
> public void join(AuthRequest authRequest) { ... }
> ```

> [!tip] 배운 점
> - Caffeine은 Redis 없이 JVM 메모리에서 동작하므로 별도 인프라 없이 즉시 적용 가능
> - `@Cacheable`은 Spring AOP 기반이므로 **같은 클래스 내 자기 호출(self-invocation)에는 동작하지 않음** — 반드시 외부 빈에서 호출해야 캐시가 적용됨
> - `@CacheEvict`로 데이터 변경 시점에 캐시를 무효화해야 정합성이 유지됨

## 3. 증분 동기화(Incremental Sync) 구현

> [!info] 배경
> - 앱 기동 시마다 `GET /auth/users`로 전체 사용자 목록을 수신해 로컬 DB를 덮어씀
> - 사용자 수가 늘어날수록 불필요한 네트워크·DB 부하 증가
> - 베트남 현지 네트워크 불안정 환경에서 대용량 응답은 기동 시간 증가로 이어짐

> [!warning] 문제
> - 변경된 사용자가 0명이어도 전체를 재수신 (낭비)
> - `syncUsers()` 내부에서 사용자 N명에 대해 개별 `SELECT` + `INSERT/UPDATE` → **N+1 쿼리**
> - 로컬 DB에 `auth_pwd`(오프라인 로그인용 해시)가 있는데, `INSERT OR REPLACE` 시 삭제될 위험

> [!success] 해결
>
> **① DB 마이그레이션 (PostgreSQL)**
> ```sql
> ALTER TABLE wms.cm_user
>     ADD COLUMN IF NOT EXISTS updated_at VARCHAR(30)
>     DEFAULT to_char(now(), 'YYYY-MM-DD"T"HH24:MI:SS');
>
> CREATE OR REPLACE FUNCTION wms.fn_cm_user_updated_at()
> RETURNS TRIGGER AS $$
> BEGIN
>     NEW.updated_at = to_char(now(), 'YYYY-MM-DD"T"HH24:MI:SS');
>     RETURN NEW;
> END;
> $$ LANGUAGE plpgsql;
>
> CREATE TRIGGER trg_cm_user_updated_at
>     BEFORE INSERT OR UPDATE ON wms.cm_user
>     FOR EACH ROW EXECUTE FUNCTION wms.fn_cm_user_updated_at();
> ```
> - `VARCHAR` 선택 이유: ISO 8601 형식 문자열은 사전순 = 시간순이므로 `>` 비교 정확
>
> **② 백엔드 — `GET /auth/users?since=` 파라미터 추가**
> - `since` 없음 → 전체 사용자(`use_yn = 'Y'`) 반환 (최초 동기화)
> - `since` 있음 → 해당 시각 이후 변경된 사용자만 반환 (Y + N 포함)
> - 응답에 `syncedAt` 추가 → 앱이 다음 요청 시 `since=` 값으로 사용
> ```xml
> <where>
>     <if test="since == null">u.use_yn = 'Y'</if>
>     <if test="since != null">u.updated_at &gt; #{since}</if>
> </where>
> ```
>
> **③ 앱 — `AuthRepository.syncUsers()` 증분 적용**
> ```dart
> Future<void> syncUsers() async {
>   final since = await _local.getLastSyncTime(); // null이면 최초(전체)
>   final result = await _remote.fetchUsers(since: since);
>   await _local.syncUsers(result.users, isIncremental: result.isIncremental);
>   await _local.setLastSyncTime(result.syncedAt);
> }
> ```
> - 마지막 동기화 시각은 `wms_app_setting` 테이블(이미 존재)에 저장
>
> **④ 앱 — `syncUsers()` N+1 → 배치 처리**
> - 전체 동기화: `rawUpdate` 1번(일괄 비활성화) + `batch.rawInsert` N번
> - 증분 동기화: `batch.rawInsert` N번만
> - `ON CONFLICT(user_id) DO UPDATE SET` 구문으로 `auth_pwd` 컬럼 제외 → 오프라인 비밀번호 해시 보존
> ```dart
> batch.rawInsert('''
>   INSERT INTO $_table (..., auth_pwd)
>   VALUES (..., '')
>   ON CONFLICT(user_id) DO UPDATE SET
>       emp_nm = excluded.emp_nm,
>       use_yn = excluded.use_yn,
>       ...
>       -- auth_pwd 는 UPDATE 대상에서 제외
> ''', [...]);
> ```

> [!tip] 배운 점
> - `INSERT OR REPLACE`(= DELETE + INSERT)는 기존 컬럼 값을 모두 날리므로 `auth_pwd` 같은 보존이 필요한 컬럼엔 `ON CONFLICT DO UPDATE` 를 써야 함
> - ISO 8601 문자열(`YYYY-MM-DDThh:mm:ss`)은 사전순 정렬이 시간순과 동일 → VARCHAR 컬럼도 `>` 비교로 증분 필터 가능
> - Sembast(Web)는 SQLite batch가 없어 순차 upsert 유지, 대신 비활성화 로직은 메모리 비교로 처리

## 🧪 테스트 방법

### 백엔드 테스트 (Swagger or curl)

**전체 동기화 확인**
```bash
# since 없이 호출 → 전체 사용자 목록 + syncedAt 반환
curl -X GET http://localhost:7100/auth/users
```
기대 응답:
```json
{
  "success": true,
  "items": {
    "items": [ { "userId": "...", "useYn": "Y", ... } ],
    "syncedAt": "2026-05-27T14:30:00"
  }
}
```

**증분 동기화 확인**
```bash
# syncedAt 값을 since로 전달 → 그 이후 변경분만 반환
curl -X GET "http://localhost:7100/auth/users?since=2026-05-27T14:30:00"
```
기대 응답: 변경된 사용자만 포함, 변경 없으면 `items: []`

**DB에서 직접 변경 후 확인**
```sql
-- 특정 사용자 수정 → updated_at 자동 갱신 확인
UPDATE wms.cm_user SET emp_nm = '테스트' WHERE user_id = 'test01';
SELECT user_id, emp_nm, updated_at FROM wms.cm_user WHERE user_id = 'test01';
```

### 앱 테스트 (단계별)

**① 최초 동기화 확인**
1. 앱 설치 후 최초 실행
2. 백엔드 로그에서 `GET /auth/users` (since 파라미터 없음) 확인
3. 로컬 SQLite에서 사용자 목록 저장 확인:
   ```sql
   -- DB Browser for SQLite 또는 Flutter DevTools로 확인
   SELECT user_id, use_yn, synced_at FROM cm_user;
   SELECT * FROM wms_app_setting WHERE setting_key = 'cm_user_sync_time';
   ```

**② 증분 동기화 확인**
1. DB에서 사용자 1명 수정 (`UPDATE wms.cm_user SET emp_nm = '변경' WHERE user_id = 'xxx'`)
2. 앱 재실행 (종료 후 재기동)
3. 백엔드 로그에서 `GET /auth/users?since=2026-05-27T...` 형태로 호출됐는지 확인
4. 로컬 DB에서 해당 사용자만 `emp_nm`이 변경됐는지, 나머지는 그대로인지 확인

**③ 오프라인 비밀번호 보존 확인**
1. 온라인 로그인 1회 수행 (로컬 `auth_pwd` 해시 저장됨)
2. 앱 재실행하여 증분 동기화 수행
3. 서버 차단 후 오프라인 로그인 시도 → 성공해야 함
   - `auth_pwd`가 동기화로 지워지면 오프라인 로그인 불가 → 이 테스트로 검증

**④ 캐시 동작 확인 (백엔드)**
1. 백엔드 로그 레벨을 `DEBUG`로 설정 (`application-local.yaml`에 이미 설정됨)
2. 동일 사용자로 API 2번 연속 호출
3. 첫 번째 호출: SQL 쿼리 로그 출력됨
4. 두 번째 호출: SQL 쿼리 로그 없음 → 캐시 히트

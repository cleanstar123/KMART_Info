## 1. 하이브리드 로그인 구조 설계

> [!info] 배경
> - PDA는 베트남 현지에서 사용 → 네트워크가 자주 끊김
> - ERP의 TA_USERM 계정 정보를 WMS 서버가 배치로 받아오고, PDA는 WMS API와만 통신
> - 네트워크 불안정 환경에서도 로그인이 가능해야 함

> [!success] 해결 — 3계층 인증 구조
> ```
> PDA 앱
>  └─ 1단계: WMS API 호출 (온라인)
>       ├─ 성공 → 유저 정보 로컬 캐시 저장 후 로그인
>       └─ NetworkException → 2단계: 로컬 SQLite 캐시 조회 (오프라인 폴백)
>            ├─ 캐시 있음 → 로그인 성공
>            └─ 캐시 없음 → OfflineFallbackException (첫 온라인 로그인 필요)
> ```
> - 서버 인증 거부(401)는 로컬 폴백 없이 즉시 오류 반환 (보안 유지)
> - 로그인 시도마다 `wms_login_log`에 API/LOCAL 구분, 성공/실패 기록

> [!tip] 배운 점
> - NetworkException과 AuthException을 구분하는 것이 핵심
> - 서버가 401을 반환하면 로컬 폴백 금지 → 보안 허점 방지

## 2. 보안 — 비밀번호 및 개인정보 암호화

> [!info] 배경
> - PDA 기기 내부 DB에 계정 정보가 저장되므로 분실/탈취 시 데이터 노출 위험
> - 비밀번호 원문 저장 금지, 개인정보 필드도 암호화 필요

> [!success] 해결
> **비밀번호 (`auth_pwd`)** — SHA-256 단방향 해시
> - 복호화 불필요, 입력값 해시 후 저장된 해시와 비교
> - `PasswordHasher.hash(plain)` / `PasswordHasher.verify(plain, hash)`
>
> **개인정보 필드 (`emp_addr`, `tel_no`, `cel_no`, `e_mail`)** — AES-256 양방향 암호화
> - 화면 표시를 위해 복호화 필요 → AES-CBC 모드
> - IV를 매 암호화마다 랜덤 생성 → 동일 값도 매번 다른 암호문 생성
> - 저장 형식: `base64(IV):base64(암호문)`
> - `FieldEncryptor.encrypt(value)` / `FieldEncryptor.decrypt(combined)`
>
> **대소문자 구분**
> - `trim()`만 적용, `toLowerCase()` 사용 금지
> - SQLite `=` 연산자는 기본 BINARY 조합으로 대소문자 구분

> [!warning] 미결 사항
> - 현재 AES 키가 앱 내 하드코딩 상태
> - 운영 배포 전 Android Keystore 또는 flutter_secure_storage로 교체 필요

> [!tip] 배운 점
> - `auth_cd`(로그인 ID)와 `emp_nm`(직원 이름)는 다른 필드
> - ADMIN 유저의 로그인 ID는 `root` (emp_nm이 ADMIN이 아님)

## 3. 공통 예외처리 구조

> [!info] 배경
> - 로그인 외에도 재고조회, 피킹 등 모든 API 호출에서 동일한 예외처리 필요
> - 각 feature마다 중복 구현하지 않도록 공통화 필요

> [!success] 해결
> ```
> lib/core/error/
>  ├─ app_exception.dart     — sealed class (NetworkException, AuthException, ServerException, OfflineFallbackException)
>  └─ api_error_handler.dart — DioException → AppException 변환 공통 함수
> ```
> - 모든 feature의 remote datasource에서 `ApiErrorHandler.handle(e)` 호출
> - `core/error/`에 위치시킨 이유: 인프라 레이어 기술 코드 → feature와 무관

## 5. SQLite DB 마이그레이션

> [!warning] 문제
> - 기존에 `version: 1`로 생성된 DB가 기기에 남아있어 새 테이블(`wms_user`, `wms_login_log`)이 추가되지 않음
> - Hot Reload/Restart로는 DB 재생성 안 됨

> [!success] 해결
> - `version: 2`로 올리고 `onUpgrade` 추가
> - `onOpen` 콜백에 `CREATE TABLE IF NOT EXISTS` 추가 → 앱 실행 시마다 누락 테이블 자동 보완
> - 개발 중 빠른 초기화: `adb shell pm clear com.example.wms_pda_app`

> [!tip] 배운 점
> - `onCreate`는 DB 최초 생성 시에만 실행
> - `onUpgrade`는 version이 올라갈 때 실행
> - `onOpen`은 매번 실행 → `IF NOT EXISTS` 와 함께 쓰면 안전한 보완책

## 6. 유저 동기화 설계 (WMS API 개발 후 적용 예정)

> [!info] 설계 방향
> - 앱 시작 시 온라인이면 `GET /api/auth/users` 호출 → 로컬 `wms_user` 갱신
> - 삭제된 유저: `use_yn = 'N'` 처리 → 로컬 로그인도 불가
> - 신규 유저: `auth_pwd = ''`로 삽입 → 첫 온라인 로그인 성공 후 해시 저장됨
> - `workmanager` 패키지로 백그라운드 배치 동기화 추가 가능 (pubspec에 주석 처리됨)
